# AI 上下文汇总方案

## 目标

把小手机里各个软件的信息整理成可以发给 AI 的上下文包。它不是把所有内容原样塞给 AI，而是按现实时间、当前角色、软件来源、聊天场景和重要程度整理，让 AI 知道最近发生了什么、当前 char 和用户的关系进展、哪些内容需要继续记住。

这个系统默认服务于“当前正在对话的角色”。跟 A 发消息时，只发送 A 相关上下文；跟 B 发消息时，只发送 B 相关上下文。除非用户明确允许，不要把 A 的私聊、专属日记、一起听记录或情感线发送给 B。

核心原则：

- 微信 / QQ 聊天记录是最高优先级，因为小手机主要玩的就是聊天。
- 按现实时间汇总，支持今天、一日总结、最近 3 天、最近 5 天、最近 7 天、自定义时间。
- 默认角色隔离，当前角色只拿自己的跨软件上下文。
- 先给用户发送预览表，让用户看清楚会发哪些软件、多少内容、如何处理。
- 聊天保留关键原话，普通寒暄可以摘要，不默认全量塞。
- 当天日记可以较完整发送，非当天日记默认总结后发送。
- 小红书、抖音、浏览器等浏览内容作为兴趣和状态线索，总结后发送。
- 每次生成和发送给 AI 的内容都要进入后台记录，方便追踪。

## 汇总入口

建议新增一个独立入口，名称可以是：

- AI 上下文
- 记忆同步
- 发送给 AI
- 生成大总结

第一版不要乱动其他模块，只读取已有数据，生成上下文和发送预览。

## 汇总范围

### 现实时间范围

所有汇总按现实时间计算，不按剧情时间或软件内虚构时间。

| 选项 | 示例范围 | 适合场景 |
|---|---|---|
| 今天 | 2026-05-05 00:00 至 2026-05-05 23:59 | 每天开始聊天前同步 |
| 最近 1 天 | 当前时间往前 24 小时 | 连续聊天 |
| 最近 3 天 | 当前时间往前 3 天 | 短期关系变化 |
| 最近 5 天 | 当前时间往前 5 天 | 默认推荐 |
| 最近 7 天 | 当前时间往前 7 天 | 周总结 |
| 自定义 | 用户选择开始和结束时间 | 精细控制 |

默认推荐：最近 5 天。

### 聊天数量

微信 / QQ 要允许用户自己调节最近消息数量。

| 软件 | 默认 | 可选 |
|---|---:|---|
| 微信单聊 | 最近 300 条 | 100 / 300 / 500 / 1000 / 自定义 |
| 微信群聊 | 最近 200 条 | 50 / 100 / 200 / 500 / 自定义 |
| QQ 单聊 | 最近 200 条 | 100 / 300 / 500 / 1000 / 自定义 |
| QQ 群聊 | 最近 150 条 | 50 / 100 / 200 / 500 / 自定义 |

聊天数量和现实时间范围同时生效：先按当前角色和场景过滤，再按时间和数量截取。

## 角色隔离

### 默认规则

当前和哪个角色对话，就只生成这个角色的上下文包。

跟 `char A` 对话时发送：

- 全局上下文：用户最近状态、公共设定、公共待办。
- `char A` 角色状态：关系、情绪、未完成话题。
- `char A` 微信单聊。
- `char A` QQ 单聊。
- `char A` 参与的群聊片段。
- 和 `char A` 相关的日记、音乐、相册、备忘、日历、浏览内容。

不发送：

- `char B` 私聊。
- `char B` 专属日记。
- `char B` 一起听记录。
- 用户和 `char B` 的暧昧、冲突、承诺、私密设定。

### 可见性标签

建议每条数据最终整理成统一事件时带这些字段：

```ts
type ContextVisibility = 'global' | 'private_to_char' | 'shared_group' | 'user_private';

type ContextEvent = {
  id: string;
  time: number;
  sourceApp: 'wechat' | 'qq' | 'diary' | 'memo' | 'calendar' | 'browser' | 'xiaohongshu' | 'douyin' | 'gallery' | 'music' | 'theater';
  sceneType: 'single_chat' | 'group_chat' | 'diary' | 'memo' | 'calendar' | 'search' | 'browse' | 'content_feed' | 'photo' | 'listen_together' | 'song_creation' | 'story';
  participants: string[];
  relatedChars: string[];
  visibility: ContextVisibility;
  importance: 'high' | 'medium' | 'low';
  summary: string;
  originalText?: string;
  metadata?: Record<string, unknown>;
};
```

生成当前角色上下文时：

- 保留 `visibility = global`。
- 保留 `relatedChars` 包含当前角色的数据。
- 保留当前角色参与的 `shared_group`。
- 默认不保留其他角色的 `private_to_char`。
- `user_private` 只有用户明确勾选才发送。

### 群聊规则

群聊不是全量共享。只发送当前角色参与过、并且和当前对话有关的群聊内容。

| 场景 | 发送规则 |
|---|---|
| A 的单聊 | 发送 A 参与的群聊摘要和关键原话 |
| B 的单聊 | 发送 B 参与的群聊摘要和关键原话 |
| A/B/C 群聊 | 发送这个群本身的上下文和成员状态 |
| 其他群 | 默认不发送 |

## 软件分装

每个角色在每个软件里都有自己的状态，不要混成一个大杂烩。建议上下文包内部按“角色 + 软件”分装。

```text
【全局摘要】
...

【当前角色：char A】
关系状态：
待延续事项：

【char A / 微信单聊】
...

【char A / 微信群聊】
...

【char A / QQ 单聊】
...

【char A / 日记关联】
...

【char A / 音乐】
...

【char A / 浏览与内容消费】
...

【char A / 相册】
...
```

### 微信 / QQ

微信和 QQ 是主上下文来源。

默认保留：

- 最近 N 条消息。
- 用户收藏消息。
- 高情绪消息。
- 关系变化消息。
- 告白、拒绝、吃醋、冲突、和好、承诺、约定。
- 设定变更。
- 用户明确标记重要的消息。

默认摘要：

- 普通寒暄。
- 重复语气词。
- 无实质信息的连续短句。
- 表情包只带描述，不带长 base64。

建议结构：

```text
【char A / 微信单聊】
时间范围：2026-05-01 至 2026-05-05
消息范围：最近 300 条内筛选

关系状态：
- ...

关键聊天：
1. 时间：2026-05-05 21:30
   摘要：char A 提到想和用户一起听歌。
   重要度：高
   原文片段：
   用户：...
   char A：...

普通聊天摘要：
- ...

未完成话题：
- ...
```

群聊要额外记录：

- 群名。
- 当前角色在群里的发言。
- 其他成员是否直接影响当前角色。
- 是否允许带其他角色发言原文。

### 日记

日记代表用户内心，也可能是角色自己的私人记录，所以要按当天和非当天分开。

| 类型 | 处理方式 |
|---|---|
| 当天用户日记 | 可以发送较完整内容，尤其是情绪、关键原话、当前角色相关内容 |
| 非当天用户日记 | 默认总结，约 50-120 字，提取情绪、关键事件、关联 char |
| 当前角色 char 日记 | 当前角色专属，可进入该角色上下文 |
| 其他角色 char 日记 | 默认不发送给当前角色 |
| 收藏 / 高重要日记 | 可以带原文片段 |

建议字段：

- 标题。
- 日期。
- owner：用户 / char。
- 关联 char。
- 情绪。
- 摘要。
- 高重要原文片段。

### 小红书 / 抖音 / 浏览器

这类内容不逐条塞，作为用户兴趣、最近状态、世界书资料的线索。

| 来源 | 发送内容 |
|---|---|
| 浏览器 | 搜索词、访问主题、世界书资料、收藏网页 |
| 小红书 | 高频浏览主题、收藏笔记、搜索关键词 |
| 抖音 | 高频观看类型、点赞/收藏、停留较久的视频主题 |

要明确区分：

- 世界书 / 设定资料：可以作为事实参考。
- 搜索历史 / 浏览记录：只能说明用户关注过，不等于世界事实。
- 内容平台浏览：主要说明兴趣、情绪、审美、剧情偏好。

浏览器生成搜索结果的预设不要使用“用户助手 / AI 助手”作为最后一条 assistant 角色消息。浏览器搜索生成只需要系统规则 + 用户搜索词 + 世界书/最近背景。assistant 角色只适合明确要做“续写开头 / start reply with / 让模型从某句话继续”的场景，默认留空。

### 音乐

音乐是关系线的一部分，不只是播放记录。

需要整理：

- 最近播放。
- 我喜欢。
- 一起听记录。
- 当前角色一起听排行。
- 当前角色创作歌曲。
- 当前角色推荐理由。

一起听记录进入角色状态：

| char | 一起听次数 | 总时长 | 最近听的歌 | 关系意义 |
|---|---:|---:|---|---|

char 创作歌曲：

```text
【char 创作歌曲】
歌名：
归属 char：
风格：
旋律：
编曲：
歌词摘要：
完整歌词是否发送：是 / 否
```

### 相册

相册只摘要，不把图片内容写得太满。

- 图片标题。
- 标签。
- 拍摄 / 上传时间。
- 关联 char。
- 是否允许当前 char 看。
- char 评论或用户备注。

### 备忘录 / 日历

用于待办和未来事件。

| 时间 | 类型 | 内容 | 相关 char | 状态 |
|---|---|---|---|---|

角色专属待办只发送给该角色；共同待办可以进入全局上下文。

### 小剧场

小剧场按剧情事件处理。

- 当前角色参与的小剧场可发送。
- 其他角色专属小剧场默认不发送。
- 公共世界事件可以摘要进入全局。

## 发送预览表

在真正发送给 AI 前，必须给用户看预览表。

| 软件 | 范围 | 当前角色相关内容 | 数量 | 处理方式 | 预计字符 |
|---|---|---|---:|---|---:|
| 微信单聊 | 最近 5 天 | char A 私聊 | 300 条内筛选 | 关键原话 + 摘要 | 9000 |
| 微信群聊 | 最近 5 天 | char A 参与群聊 | 80 条内筛选 | 摘要 + 少量原话 | 2500 |
| QQ 单聊 | 最近 5 天 | char A 私聊 | 120 条内筛选 | 关键原话 + 摘要 | 3500 |
| 日记 | 今天 | 用户今日日记 | 1 篇 | 较完整发送 | 1200 |
| 日记 | 最近 5 天 | char A 相关旧日记 | 3 篇 | 每篇 50-120 字摘要 | 600 |
| 浏览 / 小红书 / 抖音 | 最近 5 天 | 兴趣和搜索主题 | 40 条 | 聚合总结 | 1000 |
| 音乐 | 最近 5 天 | 一起听和推荐 | 12 条 | 摘要 + 排行 | 900 |
| 相册 | 最近 5 天 | 可给 char A 看 | 6 张 | 标题 + 标签 + 备注 | 500 |

预览表下方给总计：

```text
本次将发送给 AI：约 19,200 字
当前角色：char A
汇总范围：最近 5 天
聊天消息上限：微信 300 / QQ 200
是否包含其他角色私密内容：否
```

## 上下文预算

不要只用 60000 字作为唯一门槛。建议做档位。

| 档位 | 字符预算 | 适合场景 |
|---|---:|---|
| 轻量 | 8000-12000 | 快速聊天 |
| 标准 | 15000-25000 | 默认推荐 |
| 完整 | 30000-50000 | 关系回顾 / 剧情长线 |
| 超长 | 50000 以上 | 需要确认或分包 |

默认推荐：标准档。

预算分配建议：

| 内容 | 标准档占比 |
|---|---:|
| 微信 / QQ 聊天 | 60%-75% |
| 当天日记 | 5%-10% |
| 旧日记摘要 | 3%-8% |
| 音乐 / 一起听 | 3%-8% |
| 浏览 / 小红书 / 抖音 | 3%-8% |
| 相册 / 备忘 / 日历 / 小剧场 | 3%-8% |

如果超过用户选择的预算：

1. 先减少普通聊天摘要。
2. 再压缩浏览和内容平台总结。
3. 再压缩旧日记。
4. 不轻易删除高重要聊天原话。

如果 `content.length > 60000`：

1. 阻止直接发送。
2. 提醒用户当前上下文超过 60000 字。
3. 给出三个选项：
   - 发送精简版：只保留总摘要、当前角色状态、高重要聊天、高重要时间线。
   - 分包发送：按软件或时间拆分。
   - 只发送某个 char / 某个软件。

## 分包策略

分包时也要保持角色隔离。

推荐顺序：

1. `当前角色总摘要 + 人物状态 + 高重要时间线`
2. `微信 / QQ 单聊`
3. `当前角色参与的群聊`
4. `日记 / 备忘录 / 日历`
5. `浏览器 / 小红书 / 抖音 / 相册`
6. `音乐 / char 创作 / 一起听记录`

每一包开头：

```text
这是第 X/Y 包，当前角色是 {{char}}。请先只阅读和记忆，不要总结，等我发送完所有包后再回复。
```

最后一包：

```text
所有上下文已发送完毕。现在请基于以上信息继续和我对话。
```

## 类酒馆预设结构

小手机预设应该参考 SillyTavern 的 Chat Completion Prompt Manager 思路：预设不是一整段文本，而是一组有顺序、有角色、有启用状态、有插入位置的 prompt 块。

酒馆官方文档要点：

- Chat Completion API 使用 Prompt Manager 管理 prompt 构建。
- Advanced Formatting 主要服务 Text Completion API。
- Prompt Manager 里的提示块可以选择 System、AI Assistant 或 User 角色。
- 提示块可以按顺序发送，也可以设置 In-Chat 和 Depth 插入到聊天历史中。
- assistant 角色放在最后时，含义接近让模型从这条 assistant 内容继续，而不是普通系统说明。

小手机建议预设结构：

```ts
type PromptRole = 'system' | 'user' | 'assistant';
type PromptPosition = 'top' | 'before_history' | 'in_chat' | 'after_history' | 'final_prefill';

type PhonePromptBlock = {
  id: string;
  name: string;
  enabled: boolean;
  role: PromptRole;
  position: PromptPosition;
  depth?: number;
  content: string;
};

type PhonePreset = {
  id: string;
  name: string;
  mode: 'chat' | 'browser' | 'diary' | 'music' | 'theater';
  temperature?: number;
  maxTokens?: number;
  contextDepth?: number;
  blocks: PhonePromptBlock[];
};
```

### 默认聊天预设块

```text
1. Main Prompt
   role: system
   position: top
   content: 写 {{char}} 在虚拟手机聊天里的下一条自然回复。不要复述上下文，不要替用户行动。

2. Character
   role: system
   position: before_history
   content: 当前角色卡、人设、性格、开场白、系统提示。

3. User / Persona
   role: system
   position: before_history
   content: 用户昵称、用户状态、当前关系边界。

4. Phone Context
   role: system
   position: before_history
   content: 当前角色的跨软件上下文包。

5. Chat History
   role: user / assistant
   position: history
   content: 最近聊天记录。

6. Current User Message
   role: user
   position: after_history
   content: 用户最新发送的消息。
```

默认不要添加最后一条 assistant 预填。只有用户明确设置“回复开头 / 续写前缀”时，才启用：

```text
role: assistant
position: final_prefill
content: ...
```

### 浏览器预设

浏览器不是角色聊天，不需要“用户助手 / AI 助手”预设。

浏览器搜索生成建议只使用：

```text
1. Browser System
   role: system
   content: 你是虚拟手机里的浏览器搜索页。输出严格 JSON，不要 Markdown。

2. Browser Style
   role: system
   content: 搜索结果像真实互联网内容，不要出现开发说明、example.com、phone://。

3. Browser Input
   role: user
   content: 搜索词 + 浏览器世界书 + 最近背景摘要。
```

不要默认发送：

```text
role: assistant
content: 我是 AI 助手...
```

因为这会变成“最后一条是 AI 自己发送的”，容易让模型误以为已经开始输出，或者污染搜索页生成。

## 最终发送给 AI 的格式

推荐最终 prompt：

```text
你将接收一份小手机 App 的跨软件上下文总结。请把它当作长期记忆和当前状态参考。

要求：
1. 优先遵守用户当前消息。
2. 使用总结中的事实保持连续性。
3. 不要把总结逐字复述给用户。
4. 当信息冲突时，以时间更新的记录为准。
5. 默认只使用当前角色相关上下文，不要主动提其他角色私密内容。
6. 如果缺少关键信息，先自然询问，不要编造。

当前角色：{{char}}
汇总范围：{{dateRange}}
聊天数量上限：微信 {{wechatLimit}} / QQ {{qqLimit}}

以下是上下文：

【发送预览】
...

【全局摘要】
...

【当前角色状态】
...

【跨软件时间线】
...

【各软件详细信息】
...
```

## 后台记录

每次生成或发送上下文，都记录到后台记录 / 报错。

记录字段：

- 生成时间。
- 实际发送时间。
- 当前角色。
- 时间范围。
- 字符数。
- 字符预算档位。
- 涉及软件。
- 涉及 char。
- 是否包含其他角色内容。
- 是否超过 60000 字。
- 发送预览表。
- 实际发送给 AI 的文本。
- AI 返回结果。
- 报错信息。

这样用户之后能查：

- AI 到底看到了什么。
- 哪次发送失败。
- 哪些信息被精简掉。
- 有没有误带其他角色内容。

## 实现优先级

### 第一阶段

- 新增“AI 上下文 / 生成大总结”入口。
- 支持选择当前角色。
- 支持选择现实时间范围。
- 支持设置微信 / QQ 最近消息数量。
- 汇总所有软件数据为统一 `ContextEvent`。
- 默认生成当前角色上下文包。
- 输出发送预览表。
- 输出总摘要、当前角色状态、跨软件时间线、软件分区。
- 当天日记较完整，旧日记摘要。
- 浏览器 / 小红书 / 抖音总结后发送。
- 超过预算或 60000 字提醒。
- 生成内容写入后台记录。

### 第二阶段

- 支持精简版 / 标准版 / 完整版。
- 支持按软件筛选。
- 支持按 char 筛选。
- 支持分包复制或分包发送。
- 支持当前角色群聊上下文。
- 支持用户勾选“允许包含其他角色提及”。

### 第三阶段

- 自动判断高重要聊天。
- 自动把音乐一起听、日记情绪、日历事件合并进关系状态。
- 自动生成最终 prompt。
- 预设改成类酒馆 Prompt Manager：prompt 块、role、position、depth、启用状态。
- 支持角色独立预设。

## 给下一窗口的简短交接

只做上下文汇总系统和预设结构规划，不要乱动其他模块。

目标是把微信、QQ、日记、备忘录、日历、浏览器、小红书、抖音、相册、音乐、小剧场等信息，按现实时间、当前角色、软件来源整理成可以发给 AI 的上下文包。

要求：

- 微信 / QQ 是最高优先级。
- 默认按当前角色隔离，不要把 A 的私聊发给 B。
- 每个角色在每个软件下单独分装。
- 支持今天、最近 1 天、最近 3 天、最近 5 天、最近 7 天、自定义时间。
- 支持微信 / QQ 最近消息数量自定义。
- 当天日记可以较完整发送，非当天日记默认 50-120 字摘要。
- 小红书、抖音、浏览器作为浏览与内容消费总结后发送。
- 音乐模块汇总最近播放、一起听排行、char 创作歌曲。
- 发送前给用户看预览表：哪个软件、多少内容、怎么处理、预计字符数。
- 内容超过预算或 60000 字时提醒用户，不要直接发送。
- 支持精简版、分包发送、按 char / 软件筛选。
- 每次生成和发送给 AI 的内容都记录到后台记录，包括报错和 AI 返回。
- 预设参考 SillyTavern Prompt Manager，使用 system / user / assistant 角色、顺序、深度和启用状态。
- 浏览器预设删除“用户助手 / AI 助手”式 assistant 预填；浏览器只用系统规则和用户搜索输入生成搜索页。
